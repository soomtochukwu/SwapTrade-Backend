# Referral System Database Schema

This document describes the database schema for the referral system implementation.

## Overview

The referral system consists of 6 main tables that work together to track referrals, manage rewards, and maintain leaderboards.

## Tables

### 1. referral_codes

Stores unique referral codes generated by users for sharing.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | INTEGER | PRIMARY KEY | Auto-incrementing ID |
| userId | INTEGER | NOT NULL, FK -> user.id | User who owns this code |
| code | VARCHAR(12) | UNIQUE, NOT NULL | The referral code (8-12 alphanumeric chars) |
| createdAt | DATETIME | DEFAULT CURRENT_TIMESTAMP | Code creation timestamp |
| expiresAt | DATETIME | NULLABLE | Code expiration timestamp |
| isActive | BOOLEAN | DEFAULT TRUE | Whether code is currently active |

**Indexes:**
- `IDX_referral_codes_code` (UNIQUE) - For fast code lookups
- `IDX_referral_codes_userId` - For querying by user

### 2. referrals

Tracks referral relationships between users.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | INTEGER | PRIMARY KEY | Auto-incrementing ID |
| referrerId | INTEGER | NOT NULL, FK -> user.id | User who made the referral |
| referredUserId | INTEGER | NOT NULL, FK -> user.id | User who was referred |
| status | VARCHAR(20) | DEFAULT 'PENDING' | PENDING, VERIFIED, ACTIVE, REWARDED, EXPIRED |
| createdAt | DATETIME | DEFAULT CURRENT_TIMESTAMP | Referral creation timestamp |
| updatedAt | DATETIME | DEFAULT CURRENT_TIMESTAMP | Last update timestamp |
| rewardedAt | DATETIME | NULLABLE | When reward was credited |
| verifiedAt | DATETIME | NULLABLE | When referral was verified |

**Indexes:**
- `IDX_referrals_referrerId` - For querying referrer's referrals
- `IDX_referrals_referredUserId` - For querying referred users
- `IDX_referrals_status` - For filtering by status
- `IDX_referrals_createdAt` - For time-based queries

**Foreign Keys:**
- `referrerId` -> `user.id` (ON DELETE CASCADE)
- `referredUserId` -> `user.id` (ON DELETE CASCADE)

### 3. referral_rewards

Stores reward transactions for successful referrals.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | INTEGER | PRIMARY KEY | Auto-incrementing ID |
| referralId | INTEGER | NOT NULL, FK -> referrals.id | Associated referral |
| amount | DECIMAL(18,8) | DEFAULT 0 | Reward amount (high precision) |
| type | VARCHAR(50) | NOT NULL | SIGNUP_BONUS, TRADING_REWARD, VOLUME_BONUS, REFERRAL_COMMISSION |
| createdAt | DATETIME | DEFAULT CURRENT_TIMESTAMP | Reward creation timestamp |
| creditedAt | DATETIME | NULLABLE | When reward was credited to balance |
| description | VARCHAR | NULLABLE | Optional reward description |

**Indexes:**
- `IDX_referral_rewards_referralId` - For querying by referral
- `IDX_referral_rewards_type` - For filtering by reward type
- `IDX_referral_rewards_createdAt` - For time-based queries

**Foreign Keys:**
- `referralId` -> `referrals.id` (ON DELETE CASCADE)

### 4. leaderboard_cache

Cached leaderboard data for performance optimization.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | INTEGER | PRIMARY KEY | Auto-incrementing ID |
| type | VARCHAR(50) | NOT NULL | REFERRALS, TRADING_VOLUME, TOTAL_SCORE |
| userId | INTEGER | NOT NULL | User's ID |
| score | DECIMAL(18,8) | DEFAULT 0 | User's score (high precision) |
| rank | INTEGER | DEFAULT 0 | User's current rank |
| period | VARCHAR(20) | DEFAULT 'ALL_TIME' | DAILY, WEEKLY, MONTHLY, ALL_TIME |
| createdAt | DATETIME | DEFAULT CURRENT_TIMESTAMP | Record creation timestamp |
| updatedAt | DATETIME | DEFAULT CURRENT_TIMESTAMP | Last update timestamp |
| periodStart | DATETIME | NOT NULL | Period start timestamp |
| periodEnd | DATETIME | NULLABLE | Period end timestamp |

**Indexes:**
- `IDX_leaderboard_cache_type_period_rank` - Composite index for leaderboard queries
- `IDX_leaderboard_cache_userId` - For querying user's rank
- `IDX_leaderboard_cache_unique` (UNIQUE) - Unique constraint on (type, period, userId)

### 5. waitlist_users

Manages waitlist user registrations and verification status.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | UUID | PRIMARY KEY | Unique identifier |
| email | VARCHAR(255) | UNIQUE, NOT NULL | User's email address |
| name | VARCHAR(255) | NULLABLE | User's name (optional) |
| status | VARCHAR(20) | DEFAULT 'pending' | pending, verified, invited, excluded |
| referralCode | VARCHAR(12) | NULLABLE | Referral code used during signup |
| referralSource | VARCHAR(255) | NULLABLE | Source of referral (UTM, etc.) |
| verifiedAt | DATETIME | NULLABLE | Email verification timestamp |
| invitedAt | DATETIME | NULLABLE | When user was invited from waitlist |
| createdAt | DATETIME | DEFAULT CURRENT_TIMESTAMP | Signup timestamp |
| updatedAt | DATETIME | DEFAULT CURRENT_TIMESTAMP | Last update timestamp |

**Indexes:**
- `IDX_waitlist_users_email` (UNIQUE) - For email lookups
- `IDX_waitlist_users_status` - For filtering by status
- `IDX_waitlist_users_referralCode` - For referral tracking
- `IDX_waitlist_users_createdAt` - For time-based queries

### 6. waitlist_verification_tokens

Manages email verification tokens for waitlist signups.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| id | INTEGER | PRIMARY KEY | Auto-incrementing ID |
| email | VARCHAR(255) | NOT NULL, FK -> waitlist_users.email | User's email |
| token | VARCHAR(64) | UNIQUE, NOT NULL | Secure verification token |
| createdAt | DATETIME | DEFAULT CURRENT_TIMESTAMP | Token creation timestamp |
| expiresAt | DATETIME | NOT NULL | Token expiration (72 hours from creation) |
| isUsed | BOOLEAN | DEFAULT FALSE | Whether token has been used |
| usedAt | DATETIME | NULLABLE | When token was used |

**Indexes:**
- `IDX_waitlist_tokens_token` (UNIQUE) - For token validation
- `IDX_waitlist_tokens_email` - For querying by email
- `IDX_waitlist_tokens_expiresAt` - For cleanup of expired tokens

**Foreign Keys:**
- `email` -> `waitlist_users.email` (ON DELETE CASCADE)

## Entity Relationships

```
User (1) -----> (N) ReferralCode
User (1) -----> (N) Referral (as referrer)
User (1) -----> (N) Referral (as referred)
Referral (1) -----> (N) ReferralReward
WaitlistUser (1) -----> (N) WaitlistVerificationToken
```

## Data Types and Precision

### Decimal Precision
All monetary values use `DECIMAL(18,8)` precision to match the existing `UserBalance` entity pattern:
- 18 total digits
- 8 decimal places
- Allows values up to 99,999,999.99999999

### UUID vs Integer IDs
- Primary user-facing entities (waitlist_users) use UUID for security
- Internal relationship tables use INTEGER for performance
- Follows existing Trade entity patterns for UUID usage

### Timestamps
- All timestamps use DATETIME format
- Default to CURRENT_TIMESTAMP for created fields
- Nullable for optional tracking fields (verifiedAt, rewardedAt, etc.)

## Migration Strategy

The migration file `1743000000000-CreateReferralSystem.ts` creates all tables, indexes, and foreign keys in a single transaction to ensure data integrity.

### Rollback Plan
If migration fails:
1. All tables are dropped in reverse order
2. Foreign keys are dropped before tables
3. Indexes are automatically dropped with tables

## Performance Considerations

### Indexing Strategy
Following the Trade entity pattern:
- Foreign key columns are indexed for JOIN performance
- Status columns indexed for filtering
- Timestamp columns indexed for time-range queries
- Composite indexes for common query patterns (leaderboard)

### Unique Constraints
- Referral codes have unique constraint at database level
- Waitlist emails have unique constraint to prevent duplicates
- Leaderboard entries have composite unique constraint

## Security Features

### Token Management
- Verification tokens are 64 characters (secure random generation)
- Tokens expire after 72 hours
- Used tokens are marked but retained for audit purposes

### Foreign Key Constraints
- All relationships use ON DELETE CASCADE
- Prevents orphaned records
- Maintains referential integrity

## Usage Examples

### Generate Referral Code
```typescript
const code = await referralService.generateCode(userId);
// Creates entry in referral_codes table
```

### Track Referral
```typescript
await referralService.trackReferral(referrerId, referredUserId, referralCode);
// Creates entry in referrals table
```

### Award Reward
```typescript
await referralService.awardReward(referralId, amount, rewardType);
// Creates entry in referral_rewards table
```

### Update Leaderboard
```typescript
await leaderboardService.updateScore(userId, score, period);
// Upserts entry in leaderboard_cache table
```

### Verify Waitlist Email
```typescript
await waitlistService.verifyEmail(email, token);
// Validates token, updates waitlist_users.status to 'verified'
```
